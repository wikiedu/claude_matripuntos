# v3.1 Mobile-First Responsive — Implementation Plan

> Spec: `docs/superpowers/specs/2026-05-12-mobile-pwa-strategy-design.md` (Fase 1)
> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`. Despachar 5 subagentes de área en paralelo + 1 consolidador secuencial al final.

**Goal:** Reescribir todas las pantallas críticas de Matripuntos para 375×667 como breakpoint primario (mobile-first). Hoy la cobertura responsive es de 41 utilidades Tailwind en 175 archivos — esto debe pasar a > 400. Touch targets ≥ 44 px, inputs sin zoom iOS, bottom sheets en lugar de modales centrados.

**Architecture:** 5 subagentes (A-E) trabajan en áreas disjuntas en branches separadas. Cada uno entrega un PR contra `feature/v3.1-mobile-first`. Subagente F consolida los 5 PRs en la branch principal, resuelve solapes, ejecuta QA two-account y crea el PR final contra `main`.

**Tech Stack:** Tailwind 3.3, React 18, Playwright (E2E + visual snapshots), Lucide React, Zustand.

**Tag final:** `v3.1` tras merge.

**Branch base:** `feature/v3.1-mobile-first` (creada de `main` post-v3.0).

**Pre-requisito:** Fase 0a completada (CF Pages activo, Render Starter activo). Sin esto, las previews por PR no funcionan y la velocidad de iteración cae.

### Tarea 0.5 · Documento de estándares para subagentes

Crear un documento que CADA subagente debe leer antes de empezar.

- [ ] **Paso 1:** Crear `docs/internal/v3.1-mobile-standards.md` con:

```markdown
# v3.1 Mobile-first standards (para subagentes)

## Reglas absolutas

1. **Mobile-first:** clases sin breakpoint = mobile (375). Usa `md:` para ≥ 768. Olvida `sm:` excepto casos específicos.
2. **Touch target:** TODO botón, link clickable, checkbox, switch debe tener ≥ 44×44 px. Usa `min-h-touch min-w-touch` o `p-3` mínimo.
3. **Inputs sin zoom iOS:** `text-base` o más (≥ 16 px). NUNCA `text-sm` (= 14 px) en `<input>`, `<textarea>`, `<select>`.
4. **InputMode + EnterKeyHint:**
   - email → `inputMode="email" enterKeyHint="next"`
   - tel/numérico → `inputMode="numeric"` o `"decimal"`
   - search → `inputMode="search" enterKeyHint="search"`
   - último campo de form → `enterKeyHint="done"` o `"send"`
5. **Modales → bottom sheets en mobile:** Si el componente es `<Modal>` con backdrop centrado, sustituir por `<BottomSheet>` en `< md`. Mantener modal en desktop.
6. **Scroll into view en focus:** Inputs largos (`textarea`, multi-step forms) deben hacer `scrollIntoView({ block: 'center', behavior: 'smooth' })` en `onFocus`.
7. **Eliminar overflow-x:** Cualquier layout que produzca scroll horizontal en 375 px es bug crítico.
8. **Safe-area:** Bottom nav y modales fixed-bottom: `pb-safe-bottom`. Headers fixed-top: `pt-safe-top`. Ya hay tokens en Tailwind config.
9. **Tipografía:** Headers `text-2xl md:text-3xl`. Subtítulos `text-base md:text-lg`. Body `text-sm md:text-base` EXCEPTO inputs.
10. **Gestures-friendly:** Si hay swipe-to-delete o pull-to-refresh planeado en Fase 3, deja `touch-action: pan-y` en contenedores apropiados.

## Patrón de PR

- Branch: `feature/v3.1-area-<letter>-<short-name>` (ej: `feature/v3.1-area-a-dashboard`).
- PR contra `feature/v3.1-mobile-first`, NO contra `main`.
- Título: `v3.1 · Area X · <descripción>`.
- Cada PR DEBE incluir Playwright snapshots before/after en los 4 viewports.

## Tests obligatorios

Para cada componente tocado:
1. Vitest unit (si tenía tests, mantenerlos pasando).
2. Playwright snapshot a 375×667 y 1280×800 (`page.screenshot({ fullPage: true })`).
3. Manual two-account: una pantalla compartida en iPhone SE viewport y otra en desktop, verificar paridad funcional.

## Mensajes de commit

- `feat(mobile/<area>): <component> mobile-first redesign`
- `fix(mobile/<area>): <issue> in <component>`
- `test(mobile/<area>): snapshots for <component>`

## Definition of Done de un subagente

- [ ] Todos los archivos asignados refactorizados.
- [ ] Sin scroll horizontal en 375 px en ninguna pantalla del área.
- [ ] Touch targets verificados con Playwright `expect(...).toHaveCSS('min-height', '44px')` o equivalente.
- [ ] Snapshots Playwright actualizados y commited en `__screenshots__/`.
- [ ] PR abierto contra `feature/v3.1-mobile-first` con descripción detallada.
- [ ] CI verde (lint + type-check + tests).
```
