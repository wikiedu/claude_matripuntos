# v3.0 Infra Quick Wins — Implementation Plan

> Spec: `docs/superpowers/specs/2026-05-12-mobile-pwa-strategy-design.md` (Fase 0a)
> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrar frontend de FTP a Cloudflare Pages (CDN global gratis + previews por PR) y subir backend Render Free→Starter ($7/mes, mata cold start). Sin tocar BD ni código aplicación. Pre-requisito para Fase 1+2.

**Architecture:** Dos cambios paralelos e independientes. Frontend: nameservers de `matripuntos.com` se mueven de keepitup.io a Cloudflare; Cloudflare Pages se conecta al repo GitHub y auto-deploya `src/frontend/` desde `main`. Backend: upgrade en dashboard Render, sin cambio de URL. Coexistencia FTP+CF durante propagación DNS (24-48h) sirviendo mismo build.

**Tech Stack:** Cloudflare Pages, Render dashboard, DNS, GitHub Actions (deshabilitar FTP workflow al final).

**Tag final:** `v3.0` en `main` tras merge.

**Branch:** `feature/v3.0-infra-quickwins`

## Bloque C · Cloudflare Pages setup (1h)

Pre-requisito: TTL DNS ya reducido a 300 s (Tarea A.2) hace ≥ 1h.

### Tarea C.1 · Añadir matripuntos.com a Cloudflare

- [ ] **Paso 1:** Cloudflare Dashboard → Add a site → `matripuntos.com` → plan Free.
- [ ] **Paso 2:** CF escanea registros DNS existentes en keepitup. Verificar que importa correctamente:
  - `A matripuntos.com → <IP keepitup>` (proxied OFF inicialmente)
  - `CNAME www → matripuntos.com` (proxied OFF inicialmente)
  - Cualquier MX que tengas para email (proxied OFF, siempre).
- [ ] **Paso 3:** CF muestra 2 nameservers asignados, ejemplo `aria.ns.cloudflare.com` y `kurt.ns.cloudflare.com`. Anotarlos.

⚠️ NO cambiar nameservers todavía. Lo haremos al final del Bloque C tras tener CF Pages funcionando.

### Tarea C.2 · Conectar Cloudflare Pages al repo GitHub

- [ ] **Paso 1:** CF Dashboard → Workers & Pages → Create application → Pages → Connect to Git → autorizar GitHub.
- [ ] **Paso 2:** Seleccionar repo `wikiedu/claude_matripuntos`. Producción branch: `main`.
- [ ] **Paso 3:** Build settings:
  - Framework preset: **None** (manual)
  - Build command: `cd src/frontend && npm ci && npm run build`
  - Build output directory: `src/frontend/dist`
  - Root directory: (vacío, raíz)
  - Environment variables (build):
    - `NODE_VERSION` = `20`
    - `VITE_API_URL` = `https://matripuntos-api.onrender.com`
    - Cualquier `VITE_SENTRY_DSN`, `VITE_POSTHOG_KEY` que tengas en `.env.production` actual. Copiar de allí.

- [ ] **Paso 4:** Guardar y disparar el primer deploy. CF asigna una URL `matripuntos-XXX.pages.dev`.

### Tarea C.3 · Verificar deploy preview funciona

- [ ] **Paso 1:** Abrir la URL `matripuntos-XXX.pages.dev` en navegador. Debe cargar la app, login funcional, llamadas a API funcionan (van a `matripuntos-api.onrender.com`).

- [ ] **Paso 2:** Probar `/api/health` desde la consola del navegador.

```js

```

Si falla por CORS, verificar que `FRONTEND_URLS` en Render incluye el dominio `*.pages.dev` o el dominio canónico. Si no, **PAUSAR** y avisar al usuario que hay que añadir `https://matripuntos-XXX.pages.dev` a la env var `FRONTEND_URLS` en Render dashboard antes de continuar.

- [ ] **Paso 3:** Lighthouse sobre la URL CF Pages.

```bash

  --preset=desktop --output=json --output-path=/tmp/lh-cf.json --chrome-flags="--headless"
```

Esperado: Performance ≥ 80 (puede ser bajo aún, mejoraremos en Fases 1-3).

### Tarea C.4 · Configurar dominio personalizado en CF Pages

- [ ] **Paso 1:** En el proyecto CF Pages → Custom domains → Set up a domain → `matripuntos.com`.
- [ ] **Paso 2:** CF añadirá un CNAME en la zona DNS apuntando a `matripuntos-XXX.pages.dev`. Confirmar.
- [ ] **Paso 3:** Igual para `www.matripuntos.com`.
- [ ] **Paso 4:** Esperar que CF emita el certificado SSL (Universal SSL, 1-5 min normalmente).

### Tarea C.5 · Activar el proxy CF en los registros DNS

Importante: una vez nameservers cambien (siguiente tarea), los records DEBEN estar con proxy ON (nube naranja) para que CF Pages reciba el tráfico.

- [ ] **Paso 1:** Editar record `matripuntos.com A` → proxied = **ON** (orange cloud).
- [ ] **Paso 2:** Editar record `www CNAME` → proxied = **ON**.
- [ ] **Paso 3:** Verificar registros MX (email) están proxied = **OFF**. Críticamente importante — si están ON, romperás el email.

## Rollback si algo falla

**Fallo CF Pages tras switch DNS:**
1. Verificar último deploy en CF Pages dashboard → Rollback a deploy anterior.
2. Si CF Pages está caído (improbable): emergencia DNS — keepitup.io → revertir nameservers a los originales. Propagación 1-24h. Durante ese tiempo, usar `bash deploy-frontend.sh` (mantenido como rollback) para servir el último build.

**Fallo Render Starter:**
1. Dashboard Render → Plan → revertir a Free. Riesgo: vuelven los cold starts pero el servicio funciona.

**Pérdida de email durante propagación:**
1. Records MX proxied=ON por accidente → editar y poner OFF. Email se recupera al propagar (1-4h).
