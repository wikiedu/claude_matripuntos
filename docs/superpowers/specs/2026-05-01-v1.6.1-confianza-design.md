# v1.6.1 · Confianza — Design Spec

> Estado: aprobado por usuario 2026-05-01, pendiente de plan de implementación  
> Branch destino: `feature/v1.6.1-confianza`  
> Tag al mergear: `v1.6.1`  
> Release anterior: `v1.6 · La Personalidad` (en producción tras tag, ver ROADMAP)  
> Pre-requisito: `v1.5.1` (Supabase migrations reconcile) aplicado

---

## 1. Objetivo

Convertir Matripuntos en una app pública seria y mantenible: privacy legal cumplimentada, telemetría real para validar hipótesis, onboarding completo del invitado, QA automatizado E2E con Playwright, contract testing back↔front extendido y rate-limiting granular. v1.6.1 no introduce features de producto pero quita riesgos críticos antes de empujar a más usuarios y consolida la base sobre la que descansarán v1.7 y v2.x.

**Principios:**
- **Test-first.** Cada artefacto nuevo entra con tests verdes (unit + integración + E2E donde aplique).
- **Conexión back↔front impecable.** Schemas zod compartidos en workspace `packages/shared`. Una sola fuente de verdad para validación + tipos.
- **Security-by-default.** Rate-limit aplicado por defecto; PII filtrada en telemetría; cookie consent gobierna analytics.
- **GDPR pragmático nivel intermedio.** Cumplimiento real (RAT + DPAs + procedimiento ARCO-POL) sin caer en burocracia certificada empresarial.
- **Cero fricción para el usuario que ya está.** Cookie banner no bloquea; account deletion es deliberado pero claro; salir de pareja conserva el histórico personal.

## 2. Alcance aprobado

### In-scope

**Privacy & legal stack** (cumplimiento intermedio):
- Páginas legales `/privacy`, `/terms`, `/cookies` renderizadas desde markdown versionado en `docs/legal/`.
- Cookie consent banner con 3 acciones (Aceptar todo · Solo esenciales · Personalizar) y persistencia en cookie `mp_consent_v1` (TTL 12 meses).
- Borrar cuenta con anonimización del histórico (user-fantasma `__deleted-{coupleId}`).
- Salir de pareja con `Couple.dissolvedAt` + reasignación a couples individuales nuevos + histórico read-only "Mi etapa con [nombre]".
- Retención diferenciada por tipo de dato + cron diario que aplica purgas.
- Documentación legal: RAT, DPAs checklist con sub-procesadores, procedimiento de respuesta a derechos ARCO-POL.

**Telemetría producto:**
- PostHog Cloud EU (free tier hasta 1M eventos/mes).
- Wrappers `telemetry.ts` en frontend y backend con sanitize PII (whitelist).
- Catálogo cerrado de eventos canónicos en `packages/shared/telemetry-events.ts` con validación zod.
- Modo opt-out con tracking anónimo por defecto: PostHog se inicializa siempre, pero solo identifica al user (`identify`) cuando consent activo. Tracking anónimo agregado mantiene KPIs sin sesgo.
- 10 KPIs/dashboards iniciales (activación, mood adoption, phrase reach, cascade health, negotiation health, disputed rate, retention D1/D7/D30, consent rate, ratelimit hit rate, funnel signup→primera-actividad).

**Onboarding del invitado:**
- Tras `StepJoinAccount` (v1.4.1) se añaden 2 pasos ligeros: avatar (con `AvatarPicker` v1.6) + workMode/horas.
- `ProfileCompletionBanner` en dashboard durante 7 días si perfil <80%; dismiss persiste en `localStorage`.
- Endpoint `GET /api/profile/completion` devuelve `{ percent, missing[] }`.

**E2E con Playwright (15-25 specs):**
- Setup en `e2e/` raíz con projects chromium + webkit.
- Backend dockerizado con SQLite efímero (`DATABASE_URL=file:./test-${runId}.db`) en puerto 3001 + frontend dev server en 5174 vía fixture global.
- Helper `createCouple({ users: 2 })` que devuelve cookies/tokens. DB limpia por test.
- 22 specs cubriendo auth, onboarding, activity, task, mood, phrase, avatar, settings, privacy, smoke.
- CI bloquea merge si E2E falla. Tiempo objetivo <12 min con paralelización.
- Trace + screenshot + video on failure como artifact.

**Contract testing back↔front extendido:**
- Workspace nuevo `packages/shared` con schemas zod migrados desde backend + tipos derivados via `z.infer`.
- 6 contract test files (`auth`, `profile`, `events`, `tasks`, `notifications`, `points`) en `src/backend/tests/contracts/`.
- Espejo en frontend que valida que mocks de tests cumplen los mismos schemas.

**Rate-limiting granular:**
- Buckets nombrados: `authBucket` (10/min IP), `profileMutationBucket` (30/min user), `writeBucket` (60/min user), `readBucket` (200/min user), `criticalBucket` (3/hora user, para account-delete y couple-leave).
- KeyGenerator usa `userId || ip`.
- Hit dispara `telemetry.track('ratelimit.hit', { endpoint, bucket })`.
- Storage en memoria (Render single-instance); migración a Redis diferida.

### Out-of-scope (diferido — ver §10 backlog)

- Sesiones grabadas (replay) en PostHog.
- Feature flags PostHog.
- OpenAPI generado desde shared schemas.
- Visual regression (Percy/Chromatic).
- Rate-limit con Redis multi-instancia.
- Export de datos personales JSON (portabilidad GDPR).
- SMTP transaccional para email del código de borrado (alternativa dev: log consola).
- Dark pattern audit externo.

## 4. Privacy & Legal Stack — diseño detallado

### 4.1 Páginas legales

Tres páginas públicas accesibles sin auth:

```
/privacy   →  docs/legal/privacy.md     (qué datos, finalidades, base legal, sub-procesadores, derechos, contacto)
/terms     →  docs/legal/terms.md       (condiciones, código de conducta, eliminación, responsabilidad, ley aplicable España)
/cookies   →  docs/legal/cookies.md     (tabla de cookies: esenciales JWT/idioma + analítica PostHog)
```

Componente `LegalPage.tsx` reusable que toma slug y renderiza markdown con `react-markdown` + estilos warm+dark.

Footer global en todas las pantallas autenticadas y en login con links a las 3.

### 4.2 Cookie consent banner

Componente `src/frontend/src/components/CookieConsentBanner.tsx` montado en root layout:

- Banner inferior NO bloqueante (la app funciona aunque no se responda).
- Copy breve: "Usamos cookies esenciales para que la app funcione. Las analíticas son opcionales y nos ayudan a mejorarla."
- 3 botones: **Aceptar todo** · **Solo esenciales** · **Personalizar**.
- Modal de personalización con 2 toggles: **Esenciales** (fijo on, tooltip "Necesarias para login y preferencias") · **Analíticas** (toggle libre).
- Persistencia: cookie `mp_consent_v1` con `{ analytics: bool, version: 1, acceptedAt: ISO }`. TTL 12 meses → al expirar reaparece banner.
- Hook `useConsent()` expone `{ consent, setConsent, hasResponded }` leyendo la cookie.
- Inicialización condicional de PostHog según `consent.analytics`.

### 4.3 Account deletion

**Frontend wizard `DeleteAccountWizard.tsx` (3 pasos):**

1. **Educativo:** explica qué pasa con tu pareja (queda sin pareja, conserva histórico read-only) y con tus datos (anonimización con user-fantasma "Usuario eliminado" en historial). Botones "Cancelar" / "Continuar".
2. **Confirmación con password + email:** input password + botón "Enviar código por email". Al pulsar, llama `POST /api/account/delete-request { password }` que dispara email con código de 6 dígitos. Si SMTP no configurado en dev → endpoint devuelve `{ codeViaConsole: true }` y se loguea el código a `console.log` (visible en logs de Render dev/staging).
3. **Confirmación final:** input código + input texto "ELIMINAR" en mayúsculas. Botón rojo "Eliminar cuenta definitivamente". Llama `POST /api/account/delete { password, code }`.

**Backend `POST /api/account/delete`** ejecuta en transacción:

1. Valida password + código (TTL 15min, single-use).
2. `User.deletedAt = now()` (soft delete inicial).
3. Marca `User.email = null`, `User.passwordHash = null`, `User.name = 'Usuario eliminado'`.
4. Crea o reutiliza el user-fantasma `__deleted-{coupleId}` (un fantasma único por couple).
5. Reasigna referencias: `PointsTransaction.userId`, `Negotiation.proposedBy/respondedBy`, `TaskLog.completedBy/verifiedBy`, `Event.createdBy` apuntan al fantasma.
6. `MoodLog`, `Notification`, `UserProfile` del user → DELETE.
7. Si era el último user activo del `Couple` → `Couple.dissolvedAt = now()`.
8. Telemetría server-side: `account.deleted` (anónimo).
9. Tras 30 días el cron de retención hard-purga el `User` con `deletedAt` (papelera).

**Servicio:** `src/backend/src/services/accountDeletionService.ts` orquesta la transacción. Idempotente: si el user ya tiene `deletedAt` set, devuelve 200 sin reejecutar.

### 4.4 Salir de pareja

**Frontend wizard `LeaveCoupleWizard.tsx` (2 pasos):**

1. **Educativo:** ambos quedáis sin pareja activa, podréis ver "Mi etapa con [nombre]" como histórico read-only en Settings, podréis emparejaros con otra persona después.
2. **Confirmación con password.**

**Backend `POST /api/couple/leave`:**

1. Valida password.
2. `Couple.dissolvedAt = now()`.
3. Crea 2 `Couple` nuevos individuales (uno por cada user activo del couple disuelto), cada uno con su `secretKey` regenerado via `crypto.randomBytes`.
4. Reasigna `User.coupleId` al nuevo couple individual.
5. El couple antiguo queda con todos sus datos históricos accesibles solo via `GET /api/history/past-couples`.
6. Telemetría server-side: `couple.left`.

**Frontend ampliación de Settings > Pareja:**
- Sección nueva "Etapas anteriores" si `pastCouples.length > 0`. Accordion read-only por cada `dissolvedAt`-couple con: nombre del partner, fechas inicio/fin, balance final, contadores agregados (eventos, tareas).
- Botón "Salir de la pareja" abre `LeaveCoupleWizard`.

### 4.5 Documentación legal

Archivos versionados en git bajo `docs/legal/`:

- `privacy.md` — política de privacidad redactada en castellano peninsular, género neutro, lenguaje claro.
- `terms.md` — términos de uso.
- `cookies.md` — política de cookies con tabla.
- `rat.md` — Registro de Actividades de Tratamiento siguiendo plantilla AEPD: identificación responsable, fines, base legal, categorías de interesados, categorías de datos, destinatarios, plazos, transferencias internacionales, medidas técnicas/organizativas.
- `dpa-checklist.md` — checklist con un row por sub-procesador: Render (DPA online auto-firmable), Supabase (DPA online), PostHog (DPA online EU), Sentry (DPA online), FTP host (revisar contrato). Cada row con campos: status (pendiente/firmado), fecha, link al PDF.
- `arco-pol-procedure.md` — procedimiento interno de respuesta a derechos ARCO-POL en <30 días: email dedicado `privacidad@matripuntos.app` (alias), template de respuesta, log de solicitudes.
- `retention.md` — tabla de retención justificada (ver §4.6).
- `posthog-dashboard.md` — link compartido al dashboard de PostHog con KPIs y notas.

### 4.6 Retención (cron diario)

Job `src/backend/src/jobs/dataRetentionJob.ts` ejecutado diariamente:

| Tipo de dato | Retención | Justificación |
|---|---|---|
| `User`, `UserProfile`, `Couple` activos | Vida de la cuenta | Core del producto |
| `User` con `deletedAt` set | 30 días papelera | Recuperación tras error humano |
| `PointsTransaction`, `Event`, `TaskLog` | Vida de la cuenta (histórico) | Es el balance que el user quiere conservar |
| `MoodLog` | 90 días rolling | Sirve para `MyMoodWeek`; no necesita vivir más |
| `Notification` | 60 días rolling | Solo se usan en feed reciente |
| `Invitation` con `status='pending'` | 14 días tras `expiresAt` | Limpieza |
| Sentry logs | 30 días (default plan free) | Sentry lo gestiona |
| PostHog events | 12 meses (config en proyecto) | Análisis longitudinal |
| Backups Supabase | 7 días daily, 30 días weekly | Recuperación |

**Salvaguarda dry-run:** las primeras 2 ejecuciones del cron en producción corren en `DRY_RUN=true` — solo loguean qué borrarían. Tras revisar logs, se desactiva el flag.

## 5. Telemetría — diseño detallado

### 5.1 Stack

- Frontend: `posthog-js` instalado en `src/frontend`.
- Backend: `posthog-node` instalado en `src/backend`.
- Cloud: PostHog Cloud EU (Frankfurt, datos en EU). DPA firmado.
- Env vars: `VITE_POSTHOG_KEY`, `VITE_POSTHOG_HOST`, `POSTHOG_KEY`, `POSTHOG_HOST`.

### 5.2 Wrappers

`src/frontend/src/services/telemetry.ts`:

```ts
export const telemetry = {
  init: (consent: boolean) => { /* posthog.init con opt_out_capturing_by_default: !consent, persistence: 'localStorage+cookie' */ },
  identify: (userId: string, traits?: Record<string, any>) => { /* solo si consent.analytics === true */ },
  track: (event: TelemetryEvent, props?: Record<string, any>) => { /* siempre llama; PostHog respeta opt_out internamente */ },
  optIn: () => posthog.opt_in_capturing(),
  optOut: () => posthog.opt_out_capturing(),
  reset: () => posthog.reset(),  // al logout
}
```

Equivalente backend en `src/backend/src/services/telemetry.ts` con `posthog-node`. Método `track(distinctId, event, props)` con flush en shutdown.

### 5.3 PII guardrail

Función `sanitize(props)` en ambos wrappers que:
- **Whitelist:** solo deja pasar las propiedades explícitamente declaradas en el schema zod del evento canónico.
- **Blacklist defensiva:** propiedades con keys `email`, `password`, `passwordHash`, `name`, `surname`, `secretKey`, `joinCode`, `currentMood` literal, `text`, `message`, `notes` se filtran aunque pasen whitelist por error.
- En dev tira `console.warn` si una key de blacklist se intenta enviar.

### 5.4 Catálogo cerrado de eventos canónicos

Archivo `packages/shared/src/telemetry-events.ts` con un schema zod por evento. La definición es la fuente de verdad consumida por frontend y backend.

```
auth.signup_completed              { method: 'email'|'joinCode', timeToCompleteMs: number }
auth.login_success                 {}
auth.logout                        {}
onboarding.step_completed          { step: 1..6, role: 'creator'|'invitee' }
onboarding.invitee_skipped_step    { step: number }
onboarding.invitee_banner_dismissed {}
mood.set                           { moodKey: MoodKey, source: 'header'|'card'|'nudge'|'settings' }
mood.cleared                       {}
mood.expired_seen                  {}
phrase.daily_seen                  { category: PhraseCategory }
avatar.changed                     { from: string, to: string, colorChanged: boolean }
activity.created                   { type: string, hasCompensation: boolean, pointsBase: number }
activity.responded                 { response: 'accepted'|'rejected'|'counter'|'forced', round: number }
activity.force_paid                { amount: number }
task.log_created                   { category: string }
task.log_disputed                  {}
task.log_auto_verified             {}
couple.left                        {}
account.deleted                    {}
consent.changed                    { analytics: boolean }
ratelimit.hit                      { endpoint: string, bucket: string }
```

Validación runtime: si alguien llama `telemetry.track` con un evento off-schema o props inválidas, en dev tira error loud y en prod loguea warn a Sentry.

### 5.5 Identify lifecycle

- Login con consent activo: `telemetry.identify(userId, { coupleId, role, hasCompletedOnboarding, plan: 'free' })`. Sin email, name ni surname.
- Logout: `telemetry.reset()` para evitar fuga entre sesiones.
- Consent revocado: `telemetry.optOut()` + `posthog.reset()` (borra distinct id local).

### 5.6 KPIs iniciales

10 dashboards/insights guardados en PostHog y linkados desde `docs/legal/posthog-dashboard.md`:

1. **Activación**: % de signups que completan onboarding en <7d.
2. **Mood adoption** (KPI v1.6): % WAU con ≥1 `mood.set` en la semana. Target ≥30%.
3. **Phrase reach**: % de sesiones diarias con ≥1 `phrase.daily_seen`. Target ≥80%.
4. **Cascade health**: distribución de `phrase.daily_seen.category`. Anomalía si >95% es `neutra-positivo`.
5. **Negotiation health**: ratio accepted vs rejected vs forced. Anomalía si forced >20%.
6. **Disputed rate**: % de TaskLogs con `task.log_disputed`. Anomalía si >10%.
7. **Retention D1/D7/D30** por cohorte de signup.
8. **Consent rate**: % de users con `consent.changed{analytics: true}`.
9. **Ratelimit hit rate**: salto repentino → posible abuso o bug en cliente.
10. **Funnel signup→onboarding completo→primera actividad**: drop-off por step.

### 5.7 Anti-patterns evitados

- No grabamos sesiones (replay) en v1.6.1 — riesgo PII alto, requiere masking minucioso. Diferido a backlog.
- No usamos PostHog feature flags en v1.6.1 — no hay necesidad concreta. Diferido.

## 6. Onboarding del invitado — diseño detallado

### 6.1 Flujo nuevo en signup-con-joinCode

Hoy `StepJoinAccount` (v1.4.1) hace login rápido tras joinCode y manda al dashboard. Cambia a 3 pasos:

1. **Step 1 (existente, sin cambios):** nombre + password + accept terms. Registra el user con joinCode.
2. **Step 2 (nuevo) "Tu avatar":** `AvatarPicker` (componente compartido v1.6) + opcional fijar mood inicial. Botón "Continuar" (no obliga mood).
3. **Step 3 (nuevo) "¿Cómo trabajas?":** `weeklyWorkHours` slider 0-80 + `workMode` 3 botones (presencial/remoto/híbrido). Botón "Continuar" o "Saltar".
4. Marca `User.hasCompletedOnboarding = true` y manda a dashboard.

Hijos/mascotas/categorías permanecen heredados del creador (no se vuelven a preguntar).

### 6.2 ProfileCompletionBanner

Componente `src/frontend/src/components/v2/dashboard/ProfileCompletionBanner.tsx` que aparece debajo del header durante 7 días desde `User.firstLoginAt` si hay campos clave vacíos:

```
[!] Tu perfil está al 60% — completar mejora la app para ti.    [Completar →] [×]
```

- Persistencia dismiss: `localStorage.matripuntos_profile_banner_dismissed_at` (ISO timestamp).
- Visibilidad: `(now - firstLoginAt < 7d) AND (perfil < 80%) AND (!dismissed)`.
- Tap "Completar →" lleva a `Settings > Personalidad` o `Settings > Trabajo` según primer campo missing.
- Tap "×" persiste el dismiss permanente (incluso si bajan el % luego).
- Al pasar 7 días, se descarta automáticamente sin interacción.

### 6.3 Cálculo del %

Utility `src/frontend/src/utils/profileCompletion.ts`:

| Campo | Peso |
|---|---|
| `avatarEmoji` + `avatarColor` | 10% |
| `surname` | 10% |
| `workMode` | 15% |
| `weeklyWorkHours` | 15% |
| `taskPreferencesLoves[]` ≥1 | 20% |
| `taskPreferencesDislikes[]` ≥1 | 20% |
| `profilePhotoUrl` | 10% |

Banner se muestra si `% < 80%`.

### 6.4 Endpoint

`GET /api/profile/completion` devuelve:
```json
{ "percent": 60, "missing": ["taskPreferencesLoves", "profilePhotoUrl"] }
```

Permite que el frontend pinte el porcentaje sin recalcular y que la lista `missing` priorice el deeplink del CTA "Completar".

## 8. Contract testing back↔front — diseño detallado

### 8.1 Workspace `packages/shared`

Estructura nueva:

```
matripuntos/
├── package.json                    (root, workspaces incluye packages/*)
├── packages/
│   └── shared/
│       ├── package.json            (name: @matripuntos/shared, types build)
│       ├── tsconfig.json
│       └── src/
│           ├── schemas/
│           │   ├── auth.ts         (loginSchema, registerSchema, joinCodeSchema)
│           │   ├── profile.ts      (profileUpdateSchema, MOOD_KEYS)
│           │   ├── events.ts       (eventCreateSchema, eventResponseSchema)
│           │   ├── tasks.ts        (taskLogSchema, taskCreateSchema)
│           │   ├── moodHistory.ts  (query + response)
│           │   ├── account.ts      (deleteRequest, deleteConfirm)
│           │   ├── couple.ts       (leaveSchema, pastCouplesResponse)
│           │   └── index.ts
```

## 9. Rate-limiting granular — diseño detallado

### 9.1 Middleware

`src/backend/src/middleware/rateLimiter.ts`:

```ts
import rateLimit from 'express-rate-limit'
import { telemetryBackend } from '../services/telemetry'

const keyByUserOrIp = (req: Request) => (req as any).user?.id ?? req.ip

const trackHit = (bucketName: string) =>
  (req: Request, res: Response, next: NextFunction, opts: any) => {
    telemetryBackend.track('ratelimit.hit', { endpoint: req.path, bucket: bucketName })
    res.status(opts.statusCode).json(opts.message)
  }

export const authBucket = rateLimit({
  windowMs: 60_000, max: 10, keyGenerator: req => req.ip,
  message: { error: 'Demasiados intentos, prueba en un minuto' },
  handler: trackHit('auth'),
})
export const writeBucket = rateLimit({
  windowMs: 60_000, max: 60, keyGenerator: keyByUserOrIp,
  message: { error: 'Demasiadas peticiones, prueba en un minuto' },
  handler: trackHit('write'),
})
export const readBucket = rateLimit({
  windowMs: 60_000, max: 200, keyGenerator: keyByUserOrIp,
  handler: trackHit('read'),
})
export const profileMutationBucket = rateLimit({
  windowMs: 60_000, max: 30, keyGenerator: keyByUserOrIp,
  handler: trackHit('profile'),
})
export const criticalBucket = rateLimit({
  windowMs: 3_600_000, max: 3, keyGenerator: keyByUserOrIp,
  message: { error: 'Esta acción tiene un límite de 3 por hora por seguridad' },
  handler: trackHit('critical'),
})
```

### 9.2 Aplicación por ruta

| Endpoint | Bucket | Límite |
|---|---|---|
| `/api/auth/login`, `/api/auth/register`, `/api/auth/join-couple` | `authBucket` | 10/min IP |
| `PUT /api/profile/me` | `profileMutationBucket` | 30/min user |
| `POST /api/events`, `POST /api/tasks`, `POST /api/tasks/logs`, `POST /api/events/:id/*` | `writeBucket` | 60/min user |
| `GET /api/*` (lecturas) | `readBucket` | 200/min user |
| `POST /api/account/delete-request`, `POST /api/account/delete`, `POST /api/couple/leave` | `criticalBucket` | 3/hora user |

### 9.3 Tests

- `src/backend/tests/rateLimit.test.ts` — disparar 11 logins seguidos → 11º 429. PUT mood 31 veces → 31º 429.
- E2E `ratelimit.spec.ts` (incluido en suite) — banner de error legible al usuario.

### 9.4 Storage

Memoria en proceso (default `express-rate-limit`). Apto para Render single-instance. Si crece a multi-instancia, migrar a Redis vía `rate-limit-redis` — diferido a backlog, documentado en `docs/legal/scaling-notes.md`.

## 15. Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| Migración del workspace shared rompe builds CI | Hacer migración mecánica con tests verdes pre/post. Step de validación: `npm run build --workspaces && npm run test --workspaces`. |
| User-fantasma deja registros con userId no referenciable desde frontend | Endpoint `/api/users/:id` devuelve payload mínimo `{ name: 'Usuario eliminado', isDeleted: true }` para userIds fantasma. Frontend pinta sin link clicable. |
| Cron de retención borra datos por error en producción | Dry-run mode las primeras 2 ejecuciones. Solo loguea. Activar borrado real tras revisar logs. |
| Email de código de borrado no llega (SMTP no listo) | Si `process.env.SMTP_*` ausente, fallback que loguea código a Sentry con tag privado y devuelve `{ codeViaConsole: true }` en dev/staging. En prod requiere SMTP configurado o el endpoint devuelve 503 con mensaje claro. |
| Playwright en CI demasiado lento | Paralelización por workers + sharding entre chromium/webkit. Si supera 12 min sostenido → mover smoke-only a PR + full nightly. |
| Cookie banner se ve mal en pantallas pequeñas | Tests E2E mobile viewport (`devices['iPhone 13']`) específicos. |
| Anonimización de PointsTransaction crea inconsistencias en balance del partner | Tests `accountDeletionService.test.ts` verifican que el balance del partner queda igual antes y después. |
| Workspace `packages/shared` rompe deploy FTP del frontend | Build de Vite resuelve workspaces; verificar en staging antes de FTP. |
| Migración Supabase corrupta (gotcha histórico) | v1.5.1 reconcile aplicado como pre-requisito. Si v1.6 ya entró limpio, esta migración va encima sin problema. |
