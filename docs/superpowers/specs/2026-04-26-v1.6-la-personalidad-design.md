## 1. Objetivo

Convertir Matripuntos de "herramienta de gestión" a "espacio compartido con personalidad" añadiendo tres piezas pequeñas pero presentes a diario: una **frase del día** que cambia y se siente personal, un **mood** ligero por persona y por pareja, y un **avatar consistente** en toda la app. Todo siempre opcional (la app sigue funcionando sin que nadie lo toque) y todo escrito y testeado al nivel de un release de producción.

**Principios:**
- **Mood = invitación, nunca dardo.** No hay moods hostiles ("enfadado/a"), no se usa para reproches. Para conflictos hay disputas/negociación.
- **Frase = pareja, no usuario.** Determinista por pareja+día → ambos ven la misma frase a la vez. Habla sobre la pareja como unidad.
- **Avatar = identidad, no juego.** En v1.6 es solo emoji+color (lo de hoy ampliado). La gamificación de avatares (accesorios desbloqueables, ilustraciones SVG) queda en backlog.
- **Cero fricción para el que pasa.** Si no fijas mood → no estorba. Si no miras la frase → no llama. Quien quiera, lo encuentra.
- **Calidad release equipo grande.** Tests unitarios + integración, validación zod, migraciones reversibles documentadas, QA con dos cuentas reales antes de merge.

## 2. Alcance aprobado

### In-scope

**Frases del día** (frontend + backend ligero):
- Biblioteca ampliada a **~280 frases** organizadas en **8 categorías** con tono y propósito declarados.
- **Cascada por urgencia emocional:** si la pareja tiene una disputa abierta, frase de "reconciliación"; si racha rota, frase de "ánimo"; si fin de semana, frase de "celebrar"; etc. Fallback final: categoría "neutra-positivo".
- **Determinismo por pareja+día+categoría:** ambos miembros ven la misma frase el mismo día.
- Componente `DailyPhrase` rediseñado con tipografía mejor + transición suave al cambiar día. Lectura-only (no hay favoritos, no hay co-creación en v1.6).

**Mood (estado del día) detallado:**
- Catálogo fijo de **10 moods** (`MOODS[]`), agrupados por tono (positivos/neutros/bajos/negativos no hostiles), cada uno con `key`, `emoji`, `label`, `hint`, `tone`.
- **Caducidad: 24h rolling** desde `moodUpdatedAt`. Pasadas 24h se considera vacío y la app vuelve a invitar a fijar uno.
- Header muestra mood propio en badge del avatar y mood del partner como texto bajo el nombre (cuando exista).
- Tarjeta media `MoodPairCard` en dashboard (presencia "B" del visual): ambos avatars con su mood y label, tap en tu lado abre `MoodSelectorSheet` (bottom sheet).
- Componente `MoodNudge` debajo del header solo aparece si el usuario actual no tiene mood vigente; descartable por sesión.
- Vista personal `MyMoodWeek` (7 días) en `/settings/profile`: línea horizontal con un punto por día. Solo muestra moods propios (NO los del partner).
- **Sin notificaciones automáticas al partner.** Cuando A cambia su mood, B lo ve la próxima vez que abre la app, no recibe push/notificación in-app.

**Avatares (sistema actual ampliado y consolidado):**
- Catálogo único `data/avatarCatalog.ts` con **30 emojis** (animales mayoritariamente, gender-neutral) y **12 colores** (paleta warm + cool coherente con el dark theme).
- Componente `AvatarPicker` reutilizable que reemplaza la lógica duplicada hoy en `Settings.tsx` y `StepProfile.tsx`.
- Sección "Personalidad" en `Settings`: avatar, color, mood actual.
- **Migración:** se eliminan las constantes locales `AVATAR_EMOJIS`/`AVATAR_COLORS` de Settings y `EMOJIS`/`COLORS` de StepProfile y se importan desde el catálogo único. Las viejas `MOOD_OPTIONS` (5 emojis) se reemplazan por `MOODS[]`.

**Backend:**
- Nueva tabla `MoodLog` (id, userId, coupleId, moodKey, createdAt) para histórico personal.
- Migración Prisma `20260427000000_v1_6_mood_log_and_mood_keys` que:
  1. Crea `MoodLog` con índices `(userId, createdAt)` y `(coupleId, createdAt)`.
  2. Convierte `UserProfile.currentMood` (hoy emoji string) a moodKey string mapeando los 5 valores antiguos al nuevo catálogo.
- `PUT /api/profile/me` valida `currentMood` con zod contra `MOOD_KEYS` y, si cambia, escribe transaccionalmente: update profile + insert MoodLog.
- `GET /api/profile/mood-history?days=7&tz=Europe/Madrid` devuelve histórico propio agrupado por día local del usuario.
- **Sin endpoint de mood-history del partner en v1.6.**

**Roadmap update:**
- Añadir entrada v1.6 a `docs/ROADMAP.md` (hoy salta de v1.5 directo a v1.7 lógico).
- Añadir backlog en `docs/ROADMAP.md` o `docs/DECISIONS.md` con los items diferidos (ver §9).

### Out-of-scope (diferido — ver §9 backlog)

- Frases co-creadas por la pareja (escribir las suyas).
- Frases personalizadas/sugeridas por IA.
- Favoritos/guardados de frases.
- Histórico de mood compartido (ver el del partner) y vista pareja-mood-week.
- Notificación automática al partner cuando cambias mood ("Edu está cansado").
- Sistema de avatares B (12-16 ilustraciones SVG diseñadas estilo plano warm/dark).
- Sistema de avatares C (base + accesorios desbloqueables — gancho gamificación, depende v1.7).
- Mood "compuesto" o etiquetas múltiples ("cansado pero feliz").
- Sugerencias de acción según mood ("estás cansado → te recomendamos delegar X tarea").

## 4. Frases del día — diseño detallado

### 4.1 Estructura del catálogo

Archivo: `src/frontend/src/data/dailyPhrases.ts`

```ts
export type PhraseCategory =
  | 'reconciliacion'   // hay disputa/dispute abierta o reciente
  | 'animo'            // racha rota, periodo de baja actividad
  | 'celebrar'         // fin de semana, racha alta
  | 'agradecer'        // tras semana de mucho aporte de uno o ambos
  | 'calma'            // semana cargada, varios eventos largos
  | 'animo-suave'      // lunes/inicio semana, retomar
  | 'hito'             // logro reciente desbloqueado, nivel subido
  | 'neutra-positivo'; // fallback siempre

export interface Phrase {
  id: string;            // estable, formato 'cat-001'
  category: PhraseCategory;
  text: string;          // máx 140 caracteres
  tone: 'warm' | 'playful' | 'reflective' | 'celebratory';
}

export const PHRASES: Phrase[] = [/* ~280 entradas */];
```

**Distribución objetivo (~280 frases totales):**

| Categoría | Cantidad | Por qué |
|---|---|---|
| reconciliacion | 25 | Set acotado, repetir rara vez (idealmente nunca el mismo día de disputa). |
| animo | 30 | Aplica varios días seguidos durante un bache. |
| celebrar | 35 | Sale fines de semana → ~104 días/año, necesita variedad. |
| agradecer | 30 | Ocasional, calidad alta. |
| calma | 25 | Pocas semanas/año pero importante que se sienta cuidada. |
| animo-suave | 25 | Lunes recurrente. |
| hito | 30 | Cubrir hitos típicos (nivel, logro, primer mes...). |
| neutra-positivo | 80 | Es el fallback diario, máxima rotación. |

Las frases se redactan en **castellano peninsular**, género neutro (hablar de "vosotros/la pareja/lo que estáis construyendo"), sin guiños religiosos ni asunciones culturales específicas, sin referencias a hijos (porque puede no haberlos).

### 4.2 Algoritmo de cascada y selección

Archivo: `src/frontend/src/utils/dailyPhrase.ts` (refactor del actual)

```
function pickPhraseCategory(state): PhraseCategory {
  if (state.hasOpenDisputeRecent) return 'reconciliacion';
  if (state.streakBrokenLast24h) return 'animo';
  if (state.recentMilestone) return 'hito';
  if (state.weekendDay) return 'celebrar';
  if (state.heavyWeekDetected) return 'calma';
  if (state.partnerHighContribLastWeek) return 'agradecer';
  if (state.isMonday) return 'animo-suave';
  return 'neutra-positivo';
}

function getDailyPhrase(state): Phrase {
  const category = pickPhraseCategory(state);
  const pool = PHRASES.filter(p => p.category === category);
  const seed = hashString(`${state.coupleId}-${state.dayKey}-${category}`);
  return pool[seed % pool.length];
}
```

**Determinismo:**
- `dayKey` = `YYYY-MM-DD` en TZ local del usuario (`Europe/Madrid` por default; viene del cliente).
- Hash estable: `cyrb53(string)` (32-bit, sin dependencias). Implementación inline 8-10 líneas.
- Mismo seed con misma pool → mismo índice → ambos miembros ven la misma frase.

**Datos de estado consumidos** (en cliente, sin endpoint nuevo en v1.6):
- `hasOpenDisputeRecent`: hay TaskLog con status='disputed' o evento rejected en últimas 48h. Derivable de queries existentes.
- `streakBrokenLast24h`: la racha actual es 0 y ayer >0 (lo trae `useStreaks` de v1.5).
- `recentMilestone`: nivel subido o logro desbloqueado en últimas 24h (de `useAchievements`/`useGamification`).
- `weekendDay`: `getDay() in [0, 6]` en TZ local.
- `heavyWeekDetected`: ≥4 eventos largos (>3h) o suma horas eventos >25 en últimos 7 días.
- `partnerHighContribLastWeek`: el partner puntuó >60% del total semanal.
- `isMonday`: `getDay() === 1`.

Si dos condiciones aplican, gana la primera por orden de prioridad emocional definido arriba (no es ranking dinámico).

### 4.3 Componente y estilo

`src/frontend/src/components/v2/dashboard/DailyPhrase.tsx` (refactor del existente)

- Tipografía mejorada: serif italic (`font-serif italic`) para que se sienta "frase", no "label".
- Card con borde discreto morado translúcido + icono pequeño según categoría (no obligatorio, opcional v1.6.1 si testing aprieta).
- Transición suave (fade 200ms) cuando cambia categoría/día (caso edge: usuario abre dashboard a las 23:59 y vuelve a las 00:01).
- Sin botones, sin gestos. Pulsación larga: nada.

### 4.4 Tests frases

`src/frontend/src/utils/dailyPhrase.test.ts`:
- `pickPhraseCategory` tabla de casos: cada combinación de booleanos retorna categoría esperada.
- `getDailyPhrase` determinismo: mismo `state` → misma frase. Mismo `dayKey` distinto `coupleId` → frases distintas (probabilísticamente; con 280 frases las colisiones puntuales son aceptables).
- Cobertura del catálogo: cada categoría tiene ≥ N frases (N según tabla §4.1).
- Longitud máxima: ningún `text` supera 140 caracteres.

Decisión deliberada: **no incluir** 😡 enfadado/a, 😠 cabreado/a, 🤬 ni equivalentes. Conflicto se canaliza por disputas/negociación, no por etiquetas pasivas.

### 5.2 Caducidad

- `currentMood` se considera vigente si `moodUpdatedAt` está dentro de las últimas **24 horas** (cálculo en cliente para visualización, autoritativo en backend al consultar).
- Pasadas las 24h, las views muestran "Sin mood hoy" y `MoodNudge` reaparece.
- Backend NO hace cleanup automático: el campo se mantiene, simplemente expira lógicamente. Esto permite ver fácilmente cuándo fue el último mood.
- En `MoodLog` cada cambio queda registrado para histórico.

## 7. Backend — modelos, migración, endpoints

### 7.1 Modelo Prisma nuevo

`src/backend/prisma/schema.prisma`:

```prisma
model MoodLog {
  id        String   @id @default(uuid())
  userId    String
  coupleId  String
  moodKey   String
  createdAt DateTime @default(now())

  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  couple    Couple   @relation(fields: [coupleId], references: [id], onDelete: Cascade)

  @@index([userId, createdAt])
  @@index([coupleId, createdAt])
}
```

`User` y `Couple` ganan relación inversa `moodLogs MoodLog[]`.

### 7.2 Migración

`src/backend/prisma/migrations/20260427000000_v1_6_mood_log_and_mood_keys/migration.sql`:

```sql
-- 1. Crear MoodLog
CREATE TABLE "MoodLog" (
  "id" TEXT NOT NULL PRIMARY KEY,
  "userId" TEXT NOT NULL,
  "coupleId" TEXT NOT NULL,
  "moodKey" TEXT NOT NULL,
  "createdAt" DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT "MoodLog_userId_fkey" FOREIGN KEY ("userId") REFERENCES "User"("id") ON DELETE CASCADE,
  CONSTRAINT "MoodLog_coupleId_fkey" FOREIGN KEY ("coupleId") REFERENCES "Couple"("id") ON DELETE CASCADE
);

CREATE INDEX "MoodLog_userId_createdAt_idx" ON "MoodLog"("userId", "createdAt");
CREATE INDEX "MoodLog_coupleId_createdAt_idx" ON "MoodLog"("coupleId", "createdAt");

-- 2. Migrar currentMood antiguo (emoji string) → moodKey nuevo
UPDATE "UserProfile" SET "currentMood" = 'feliz'      WHERE "currentMood" = '😊';
UPDATE "UserProfile" SET "currentMood" = 'tranquilo'  WHERE "currentMood" = '😎';
UPDATE "UserProfile" SET "currentMood" = 'cansado'    WHERE "currentMood" = '😴';
UPDATE "UserProfile" SET "currentMood" = 'estresado'  WHERE "currentMood" = '😰';
UPDATE "UserProfile" SET "currentMood" = 'tranquilo'  WHERE "currentMood" = '😐';
UPDATE "UserProfile" SET "currentMood" = NULL         WHERE "currentMood" IS NOT NULL
                                                       AND "currentMood" NOT IN ('feliz','enamorado','energico','carinoso','tranquilo','pensativo','cansado','enfermo','estresado','bajon');
```

**Nota Prisma+Supabase:** dada la corrupción conocida de `_prisma_migrations` desde 2026-04-10 (memoria `project_prisma_supabase_gotcha`), antes de aplicar en producción Render → Supabase, reconciliar tabla de migraciones manualmente o ejecutar el SQL directamente. ### 7.3 Endpoints

**Refactor `PUT /api/profile/me`** (`src/backend/src/routes/profile.ts`):

```ts
const profileUpdateSchema = z.object({
  // ...existentes
  currentMood: z.enum(MOOD_KEYS as [string, ...string[]]).nullable().optional(),
});

// Dentro del handler, si body.currentMood !== undefined y distinto del actual:
await prisma.$transaction(async (tx) => {
  const updated = await tx.userProfile.upsert({
    where: { userId },
    update: {
      ...allowedFields,
      currentMood: body.currentMood,
      moodUpdatedAt: body.currentMood ? new Date() : null,
    },
    create: { /* ... */ },
  });
  if (body.currentMood && body.currentMood !== existing?.currentMood) {
    // Anti-spam: solo log si último MoodLog del usuario es >5 min o distinto mood
    const lastLog = await tx.moodLog.findFirst({
      where: { userId },
      orderBy: { createdAt: 'desc' },
    });
    const fiveMinAgo = Date.now() - 5 * 60 * 1000;
    const shouldLog = !lastLog
      || lastLog.moodKey !== body.currentMood
      || lastLog.createdAt.getTime() < fiveMinAgo;
    if (shouldLog) {
      await tx.moodLog.create({
        data: { userId, coupleId, moodKey: body.currentMood },
      });
    }
  }
  return updated;
});
```

## 11. Criterios de éxito

**Funcionales:**
- Frase del día cambia coherentemente con estado de la pareja, ambos miembros ven la misma.
- Mood persiste 24h, expira limpio, MoodLog acumula histórico.
- Avatar consistente entre onboarding y settings.
- Cero regresiones en flujos existentes (auth, eventos, tareas, puntos).

**Soft (medibles tras 2 semanas en producción):**
- ≥ 30% de usuarios activos fijan mood al menos una vez por semana.
- ≥ 80% de sesiones diarias incluyen vista del DailyPhrase (renderizado en dashboard).
- Cero tickets de soporte sobre "no puedo cambiar mi mood" / "no aparece la frase".

**Anti-feature signals (revisar y corregir si aparecen):**
- Reportes de mood usado como "indirecta" para conflictos → revisar copy o quitar tone negative-soft.
- Fricción percibida del MoodNudge (queja) → reducir a banner aún más sutil o quitar.
