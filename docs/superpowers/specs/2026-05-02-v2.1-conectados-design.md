# v2.1 · Conectados — Spec

**Fecha:** 2026-05-02
**Versión:** v2.1
**Branch:** `feature/v2.1-conectados`
**Tag al merge:** `v2.1`

## 1. Goal

Sacar la app del silo: integraciones bidireccionales con servicios externos (Google Calendar, push real, ICS export) + sistema de **referidos** que premia traer parejas amigas.

## 2. Problemas a resolver

1. v2.0.1 tiene Google Calendar **read-only** — los eventos creados en Matripuntos NO suben a Google.
2. Web push v1.7 está esquemático: necesita VAPID + service worker robusto + deep links.
3. No hay forma de exportar el calendario a iCal / Apple Calendar / Outlook.
4. Crecimiento orgánico inexistente — la única manera de invitar es vía link de pareja.
5. Referidos: incentivo para traer otras parejas amigas que generan red social entre couples.

## 3. Pillars

- **Bidireccional cuando aporta valor** (calendar sync), unidireccional cuando no (push).
- **Privacy intacta**: referidos NO comparten datos de pareja, solo cuentan registros.
- **Sin gamificación tóxica**: el referido NO da puntos en la app, da perks cosméticos.
- **Estándar abierto**: ICS RFC 5545 para exportar.

## 4. Scope v2.1

### 4.1 Google Calendar bidireccional (write)
- Setting opt-in: "Crear eventos también en mi Google Calendar".
- Cuando user crea Activity → backend usa `googleapis.calendar.events.insert` con scope `calendar.events`.
- Match con metadata.googleEventId para edits/deletes posteriores.
- Conflict resolution: si user edita en Google después, en próximo sync prevalece Google (read-only-from-google policy).

### 4.2 Web push real + service worker robusto
- VAPID keys activas en producción.
- Service worker con offline caching de assets.
- Deep links: tap notification → ruta específica (`/dashboard?from=push`, `/journal#entry-id`).
- Settings → Notificaciones con 6 toggles: achievements, challenges, partner activity, mood reminder, weekly retrospective, re-engagement.
- Frequency caps por categoría (1/48h por defecto).

### 4.3 ICS export
- `GET /api/calendar/v2/export.ics` → archivo iCal con todas las CalendarEntry de los próximos/pasados 90d.
- URL pública con token firmado (no requiere auth) para suscripción tipo Apple Calendar.
- Cache CDN 1h.

### 4.4 Sistema de referidos
- Cada pareja tiene `referralCode` (8 chars alfanumérico, generado al activar premium o pasar 30d).
- URL `/r/:code` redirige a signup pre-rellenado con tracking.
- Cuando 1 user del couple referido completa onboarding → "amigos couples" se vinculan (relación opcional).
- Reward referente: 1 mes premium gratis después de 3 referidos exitosos. Reward referido: skip onboarding tutorial + badge "Recomendado por X".

### 4.5 Apple Calendar / Outlook subscriptions
- Subscribe URL signed `https://matripuntos-api.onrender.com/cal/:userToken.ics`.
- Refresh cada 6h (controlado por el cliente).
- Token revocable desde Settings.

## 6. API routes nuevas

```
POST /api/calendar/google/sync                  → bidireccional now (write enabled)
PUT  /api/calendar/google/settings              → toggle write enabled
GET  /api/calendar/v2/export.ics                → con auth, descarga
GET  /api/calendar/v2/subscribe-url             → genera token + URL pública
DELETE /api/calendar/v2/subscribe-url           → revoca

POST /api/notifications/push/settings           → 6 toggles + caps
POST /api/notifications/push/test-deep-link     → test deep link handling

GET  /api/referrals/code                        → mi código actual (genera si no existe)
POST /api/referrals/claim                       → 1 user del referido completa onboarding
GET  /api/referrals/stats                       → mi count + rewards earned

GET  /cal/:token.ics                            → público, no auth (token firmado)
GET  /r/:code                                   → redirect signup
```

## 7. Decisiones brainstorm

| # | Tema | Decisión | Razón |
|---|---|---|---|
| 1 | Google sync direction | Bidireccional opt-in, default off | Privacy + complejidad |
| 2 | Conflict resolution | Last-write-wins por timestamp; Google prevalece en read-back | Simpler |
| 3 | ICS subscription | URL pública con token firmado revocable | Estándar industria |
| 4 | Push deep links | URL params `?from=push&action=Y` | Trazable + universal |
| 5 | Referral reward | 1 mes premium tras 3 referidos | Aspiracional sin spam |
| 6 | Referral code length | 8 chars alfanumérico | Suficiente entropía + memorable |
| 7 | Referral expiry | 90 días desde generación | Evita spam viejo |
| 8 | Apple Calendar refresh | 6h | iOS default, no spam |
| 9 | Notification categories | 6 (no más para simplicidad) | Granular sin saturación |
| 10 | Write-to-google toggle | Por user (no couple) | Cada user su Calendar |

## 8. Out-of-scope

- Apple Calendar bidireccional — iCloud no expone webhook write.
- Outlook integration — niche en target ES B2C.
- WhatsApp / Telegram bots — overhead, requiere Business API.
- Slack workspace integration — wrong audience.

## 9. Riesgos

| Riesgo | Mitigación |
|---|---|
| Google quota en write | Backoff exponencial + queue persistente |
| Push payloads con PII en notif | Sanitize: solo nombres iniciales, no contenido completo |
| ICS subscription URL filtrada | Token revocable + log de reads + alerta si patrón anómalo |
| Referral spam (1 user creando N codes) | Cap 5 codes activos / couple |
| Apple Calendar caché agresivo | Cache headers `s-maxage=21600` |

## 10. Métricas éxito 30d

- Google sync write opt-in: 25%+
- Push opt-in: 50%+ (vs 40% baseline v1.7 sin VAPID)
- ICS subscriptions activas: ≥10% MAU
- Referrals enviados: ≥0.5/couple/mes
- Conversión referral: ≥30% (link → signup)
