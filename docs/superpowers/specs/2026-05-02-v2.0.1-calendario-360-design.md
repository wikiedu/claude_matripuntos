# v2.0.1 · Calendario 360 — Spec

**Fecha:** 2026-05-02
**Versión:** v2.0.1
**Branch:** `feature/v2.0.1-calendario-360`
**Tag al merge:** `v2.0.1`

## 1. Goal

Convertir el Calendar actual (vista limitada que solo muestra eventos de la pareja) en un **hub temporal unificado** donde los users planifican y revisan: actividades, tareas (programadas y recurrentes), servicios externos (limpieza, jardinería), cumpleaños/aniversarios, festivos locales y eventos importados de Google Calendar (read-only).

## 2. Problemas a resolver

1. **Calendar actual es ciego**: solo muestra `Event` (actividades). No hay tareas programadas ni recurrencias visibles, ni cumpleaños familiares, ni festivos.
2. **Planning manual fuera de la app**: las parejas usan Google Calendar para citas, médicos, viajes — y luego registran las "tareas Matripuntos" por separado. Doble esfuerzo.
3. **Cumpleaños de hijos/familia se olvidan**: no hay forma de marcarlos en la app aunque la pareja tiene Child registrado con dateOfBirth.
4. **Servicios externos invisibles**: la limpiadora viene martes y jueves, pero el calendario no lo muestra → la pareja no sabe si "ese día tenemos servicio externo o nos toca a nosotros".
5. **Festivos no son visibles**: no es trivial saber si "este sábado es festivo" sin buscar fuera de la app.

## 3. Pillars de diseño

- **Lectura primero**: importar/sincronizar es prioritario sobre crear desde cero.
- **Privacy-first en imports**: Google Calendar OAuth con scope mínimo (read-only events del calendar primario) + opción de import por categoría.
- **No duplicación**: un evento de Matripuntos NO se sube a Google. Solo se baja info externa para visualizar.
- **Vistas por densidad**: día (lista densa), semana (compacto), mes (heatmap visual), año (overview).
- **Filtros granulares**: tipo (event/task/service/birthday/holiday/external), persona, categoría.
- **Mobile-first**: el calendar móvil es lo que se usa.

## 4. Scope v2.0.1

### 4.1 Entradas unificadas

Reaprovechamos el modelo existente `CalendarEntry` (introducido en v1.4) y lo extendemos:

```prisma
model CalendarEntry {
  id          String   @id @default(cuid())
  coupleId    String
  couple      Couple   @relation(fields: [coupleId], references: [id], onDelete: Cascade)
  type        String   // event | task | service | birthday | holiday | external
  title       String
  description String?
  date        DateTime
  endDate     DateTime?
  allDay      Boolean  @default(true)
  category    String?
  // v2.0.1 additions:
  externalSource String?  // 'google_calendar' | 'manual' | 'auto'
  externalId     String?  // ID del provider (Google event id)
  recurrence     String?  // RRULE compatible con RFC 5545 simplificado
  color          String?  // override del color por defecto del tipo
  metadata       String?  // JSON: { childId?, serviceProviderId?, eventId? }
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([coupleId, date])
  @@index([coupleId, type])
}

model GoogleCalendarSync {
  id           String   @id @default(cuid())
  userId       String   @unique
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  refreshToken String   // encriptado AES-256-GCM
  scope        String
  syncEnabled  Boolean  @default(true)
  lastSyncedAt DateTime?
  syncWindow   Int      @default(90) // días forward + backward
  filters      String   // JSON whitelist de calendarIds
}

model ServiceProvider {
  id          String   @id @default(cuid())
  coupleId    String
  couple      Couple   @relation(fields: [coupleId], references: [id], onDelete: Cascade)
  name        String   // "Carmen — limpieza"
  type        String   // limpieza | jardineria | cuidado_ninos | mantenimiento | otro
  recurrence  String?  // RRULE
  color       String?
  active      Boolean  @default(true)
  notes       String?
}
```

### 4.2 Vistas

- **Día** (default móvil): timeline vertical, separadores por hora, badges color por tipo.
- **Semana**: 7 columnas × franjas hora, agrupado.
- **Mes**: grid 7×6, cada celda con dots por tipo (max 4 visibles + "+N").
- **Año** (premium-lite): heatmap mensual con intensidad = #entries.

Switcher persistente en `/calendar` con preferencia en localStorage.

### 4.3 Auto-derivación

Triggers automáticos que crean `CalendarEntry` sin acción del user:

- **Birthdays**: cron diario crea entries para cumpleaños de Child y aniversario de pareja (calculado desde `Couple.createdAt` o campo `anniversaryDate` opcional).
- **Holidays**: import semanal del calendario de holidays según `Couple.country` (default España). Mantenido en `data/holidays-es-2026.json` actualizado anualmente.
- **Tasks programadas + recurrencia**: cuando una `Task` tiene `scheduledFor` o `isRecurring + frequency`, el cron expande las próximas N ocurrencias como `CalendarEntry` con `metadata.taskId`.
- **Service providers**: cron expande `ServiceProvider.recurrence` en entries.

### 4.4 Google Calendar import (opt-in)

Flow OAuth:
1. Settings → Calendario → "Conectar Google Calendar".
2. OAuth con scope `https://www.googleapis.com/auth/calendar.readonly`.
3. User selecciona qué calendars importar (whitelist).
4. Cron diario sync: `events.list` con timeMin = -30d, timeMax = +90d.
5. Mapping: external events → `CalendarEntry` con `externalSource='google_calendar'`, `externalId=event.id`.
6. Eventos editados/borrados en Google se reflejan vía `syncToken` incremental.

**No escribimos en Google nunca.** Read-only.

### 4.5 Filtros + búsqueda

- Filtros toggle: Mis cosas · Pareja · Hijos · Servicios · Festivos · Externos.
- Búsqueda full-text por title/description (LIKE simple, no FTS).
- Quick-jump: "Hoy" · "Esta semana" · "Próxima cita".

### 4.6 Crear entry desde calendar

- Botón FAB "+" → sheet con tipo + título + fecha + opciones avanzadas según tipo.
- Para `event` y `task`, redirige al flow apropiado (no duplicación).
- Para `service` y `birthday`, crea directamente en `CalendarEntry`.

### 4.7 Detalles entry

- Tap → bottom sheet con detalles + actions:
  - Si es `event`: Ver actividad, aceptar/rechazar (atajos del flow normal).
  - Si es `task`: Marcar completada, posponer, ver historial.
  - Si es `external` (Google): solo lectura + link al evento original.
  - Si es `service`: editar, marcar completado, pausar recurrencia.
  - Si es `birthday`: ver edad calculada, ideas de regalo (futuro v3.x con IA).
  - Si es `holiday`: solo lectura.

## 6. Decisiones brainstorm

| # | Tema | Decisión | Razón |
|---|---|---|---|
| 1 | Modelo unificado | Reusar `CalendarEntry` con type discriminator | Evita 6 tablas paralelas |
| 2 | Google Calendar | Read-only OAuth | Privacy + no responsabilidad de mantener sync bidireccional |
| 3 | Recurrencia | RRULE simplificado (FREQ + INTERVAL + UNTIL) | Estándar interop, no necesitamos full RFC 5545 |
| 4 | Holidays | JSON estático actualizado anualmente | Sin dep externa, simple |
| 5 | Birthdays | Auto-derive de Child + Couple anniversary | Sin user action |
| 6 | Service providers | Modelo separado + recurrence | Reusable y testeable |
| 7 | Vistas | Día/semana/mes/año, switcher persistente | Cubre 95% casos |
| 8 | Sync frequency | Daily cron Google + hourly tasks expansion | Latencia tolerable, coste bajo |
| 9 | Encriptación refresh tokens | AES-256-GCM, key en env | Standard B2C |
| 10 | Premium gating | Año view + multi-calendar Google + ICS export | Personalizable, no bloquea core |

## 7. Out-of-scope v2.0.1

- Bidirectional sync con Google → v3.x (complejo, riesgo de duplicación).
- Apple iCloud Calendar → v3.x.
- Multi-coupleshare (compartir entries con otra pareja amiga) → futuro nicho.
- AI suggestions ("también podrías agendar X") → v3+.
- Notifications de eventos próximos → integrado con v1.7 push system.

## 8. Riesgos

| Riesgo | Mitigación |
|---|---|
| Google quota OAuth | Cron sync 1/día por user, batch users |
| Refresh tokens leaks | Encriptación + rotation cada 90d |
| Recurrencia infinita en cron | Cap forward de 90d por defecto |
| Holidays desactualizados | Annual review en CHANGELOG + workflow GitHub bot que actualiza JSON |
| User cierra cuenta Google → tokens inválidos | Detect 401 → clear refreshToken + notify user |

## 9. Métricas éxito 30d

- 35%+ users conectan Google Calendar.
- 80%+ users con couple activo registran al menos 1 service provider.
- Calendar usage: 50%+ users abren `/calendar` ≥3×/semana.
- Reduction de "tareas duplicadas" reportadas en feedback: -50%.
